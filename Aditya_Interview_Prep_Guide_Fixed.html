Salesforce Marketing Cloud

SFMC · Journey Builder · Automation Studio · Data Extensions · AMPscript · REST/SOAP API
Your foundation: Cognizant (global restaurant chain, 1M+ contact records) + Horizontal Digital (SFMC migration, zero post-go-live defects). Use these as proof points in every answer.
Module 1 — SFMC Architecture & Core Concepts
Foundation
🏗
What is SFMC?
Enterprise marketing automation platform. Separate from the Salesforce core org — communicates via Marketing Cloud Connect. Used by brands to orchestrate email, SMS, push, ads, and web journeys at scale.
🔗
SFMC vs Salesforce Core (CRM)
SFMC lives in a separate cloud environment. Marketing Cloud Connect bridges the two — syncing Contacts/Leads from CRM into SFMC as Subscribers. The Subscriber Key is the shared identifier. Common QA test: verify Subscriber Key consistency across both systems.
📊
Studios & Builders Overview
Email Studio (email sends), Mobile Studio (SMS/push), Advertising Studio (social ads), Journey Builder (multi-channel orchestration), Automation Studio (scheduled/triggered jobs), Contact Builder (data relationships), Content Builder (templates).
Module 2 — Journey Builder Deep Dive
Critical for interviews
🚀
Entry Sources
How contacts enter a journey: Data Extension (scheduled query), Salesforce Data (CRM record triggers), API Event (real-time), CloudPages (form submission), Audience (manual segment). QA must verify entry criteria are precise — wrong audience = mass email blast incident.
🔀
Decision Splits & Engagement Splits
Decision Split: routes contacts based on Data Extension field values or CRM data (e.g., "Is loyalty tier = Gold?"). Engagement Split: routes based on email interaction — Opened, Clicked, Not Opened. Test both True and False paths with edge-case contacts.
⏱
Wait Activities
Wait by Duration (X days/hours), Wait until Date, Wait until Activity (event-based). Critical QA risk: testing Wait activities in non-test mode. Use Journey Builder Test Mode to bypass waits, but be aware Test Mode limits are 10 contacts max.
📧
Send Activities & AMPscript Validation
Email Send Activity uses a From Address, Email Content, and Data Extension for personalization. AMPscript is rendered server-side at send time. Test: seed the test DE with NULL values, special characters, and boundary data to catch AMPscript runtime errors before production send.
Module 3 — Automation Studio & SQL
Your strongest technical area
🔄
Automation Types
Scheduled Automation: runs at a fixed schedule (daily at 2am). Triggered Automation: fires when a file arrives in SFTP or via API. Activities: SQL Query, Data Extract, File Transfer, Import File, Send Email, Filter. QA tests: verify automation ran (Last Run Status), check activity logs, validate output DE row counts.
🗃
SQL Query Activities
SFMC uses SQL Server T-SQL syntax. Queries run against Data Views (system tables: _Open, _Click, _Bounce, _Sent, _Subscribers) and custom DEs. Common QA validation: compare SELECT COUNT(*) before vs after automation run to verify correct record movement. Watch for Overwrite vs Append vs Update mode bugs.
📋
Data Validation SQL Patterns
Count validation: SELECT COUNT(*) FROM [DE_Name]. Duplicate check: SELECT SubscriberKey, COUNT(*) FROM [DE] GROUP BY SubscriberKey HAVING COUNT(*) > 1. NULL check: SELECT * FROM [DE] WHERE EmailAddress IS NULL. These are the scripts you used for 1M+ contact validation at Cognizant.
Module 4 — Data Extensions & Contact Builder
Data architecture essentials
📂
DE Types & When to Use
Sendable DE: linked to All Subscribers via Subscriber Key or Email — used as send audiences. Non-Sendable DE: lookup/reference data (product catalog, suppression list, transaction history). Synchronized DE: auto-synced from CRM objects via Marketing Cloud Connect. QA: verify the relationship field exists and is correctly mapped.
🔑
Subscriber Key — The Critical Identifier
Every SFMC subscriber is identified by a Subscriber Key (usually Contact ID or Email). If Subscriber Keys don't match between CRM and SFMC, sends fail silently or go to wrong subscribers. QA test: cross-reference Subscriber Keys between _Subscribers Data View and the source CRM contacts list post-migration.
🗺
Contact Builder & Data Relationships
Contact Builder defines how multiple DEs relate to each other using cardinality (1:1, 1:N, N:N) linked via Contact Key. Used in Journey Builder for multi-data-source personalization. QA: verify relationship cardinality is correct and that Lookup functions in AMPscript return expected results across all relationship paths.
Module 5 — AMPscript & Personalization
Email content intelligence
✍
Core AMPscript Functions
AttributeValue('FirstName') — retrieves subscriber attribute. Lookup('ProductDE','ProductName','ProductID',@ProductID) — looks up a value from a DE. LookupRows('TransactionDE','CustomerID',@CustID) — returns multiple rows. IF/ELSE for conditional content. QA test: each function with NULL, empty string, and unexpected data type inputs.
⚠
AMPscript Runtime Errors — QA Critical
AMPscript errors suppress the send for that subscriber but don't alert loudly. Common failure modes: Lookup returns no result (returns empty string, not NULL — test for empty string handling), SET variable not initialized before use, wrong DE name (case-sensitive in some orgs). Always test with edge-case data before any production send.
Module 6 — REST/SOAP API Testing in SFMC
API integration QA
🔌
Authentication Flow
OAuth 2.0: POST to /v2/token with client_id, client_secret, grant_type. Returns access_token (valid 20 min). QA tests: expired token handling (should return 401), missing credentials (should return 400), token rotation behavior in long-running automation.
📤
Key REST Endpoints to Know
Upsert DE records: POST /data/v1/async/dataextensions/key:{key}/rows. Trigger Triggered Send: POST /messaging/v1/messageDefinitionSends/key:{key}/send. Journey entry via API Event: POST /interaction/v1/events. Fire on test contact in sandbox, validate data persisted in target DE.
🔧
SOAP API — Legacy Integrations
Used for subscriber management (create/update/delete), email sends, and list management in older SFMC implementations. WSDL-based. Tools: Postman (with SOAP body), SoapUI. QA: validate subscriber status changes (Active/Bounced/Unsubscribed) propagate correctly and reflect in _Subscribers Data View.
What Modern Projects Do with SFMC
Modern SFMC implementations use API Events for real-time journey triggering (vs batch-scheduled), integrate with CDP (Customer Data Platform) for unified contact profiles, use Einstein for AI-driven send-time optimization, implement Salesforce Interactions SDK for web/mobile behavioral tracking, and automate DE management via CI/CD pipelines calling SFMC APIs. Quality teams are expected to test API-driven journeys end-to-end, validate CDP segment sync, and script regression suites against SFMC APIs using TypeScript/Playwright.

Walk me through how you tested Journey Builder campaigns end-to-end.
▼
I approach Journey Builder QA in five phases. First, entry source validation — I verify that the Data Extension query or API Event returns exactly the intended audience segment, testing edge cases like contacts that partially match criteria. Second, path coverage — I create test contacts seeded with data that routes them through every Decision Split path including NULL/fallback paths, and verify each exits the journey correctly. Third, Wait Activity behavior — in Test Mode I bypass waits (it has a 10-contact limit), then verify in a lower environment that production wait timing is configured correctly. Fourth, content validation — I test AMPscript personalization with NULL values, special characters, and long strings to catch runtime errors that suppress sends silently. Fifth, exit and re-entry — I verify contacts exit at the right point and test re-entry settings (can same contact enter again? After how long?).

How do you validate Data Extension integrity after a migration?
▼
My DE validation process uses SQL-based verification scripts in Automation Studio against three dimensions. Volume: SELECT COUNT(*) FROM [Target_DE] vs source count — if numbers don't match, something was dropped or duplicated. Quality: NULL check on required fields, duplicate Subscriber Key check using GROUP BY HAVING COUNT(*) > 1, field data type validation. Integrity: cross-reference Subscriber Keys against the _Subscribers Data View to ensure every migrated contact exists as an active subscriber. For the Horizontal Digital migration specifically, I also validated AMPscript personalization for a sample of 500 contacts after migration to confirm that Lookup functions were returning correct values from the new DE structure.

What's the difference between a Sendable and Non-Sendable Data Extension?
▼
A Sendable DE has a defined relationship to All Subscribers via a Subscriber Key or Email Address field — it can be the direct audience for an email send or journey. It must have the relationship configured in the DE settings. A Non-Sendable DE stores lookup/reference data — product catalogues, transaction history, suppression lists, loyalty tier definitions — that you reference via AMPscript Lookup functions but never send to directly. When testing, I verify: Sendable DEs have the relationship field correctly mapped and that all Subscriber Key values actually exist in All Subscribers (missing keys cause silent send failures), and Non-Sendable DEs have the right primary key defined for efficient Lookup queries.
How do you test SFMC REST API integrations?
▼
I test SFMC APIs in Postman across four dimensions. Authentication: POST to /v2/token, verify 200 with access_token, then test with expired token (expect 401), invalid credentials (expect 401), and missing parameters (expect 400). Payload validation: send valid request, verify 200 and data persists in target DE; then test with missing required fields, wrong data types, and oversized payloads. Business logic: for API Event journey entries, verify the contact appears in the journey within expected time, validate journey data is correctly bound from the event payload. Error handling: test rate limit behavior (SFMC throttles at ~2000 requests/min for some endpoints), verify the integration handles 429 with retry logic. I document all response schemas and use Postman collection tests to automate these validations as part of regression.
Tell me about Automation Studio — what activities did you test and how?
▼
Automation Studio is SFMC's backend job scheduler. The main activities I've tested: SQL Query Activity — I validate by comparing row counts before and after run, checking the target DE for expected records, and verifying the Overwrite/Append/Update mode is correct (Overwrite when you want a clean refresh, Append when accumulating, Update when patching). Data Extract and File Transfer — verify the output file appears in the correct SFTP folder with the right naming convention. Import File — validate column mapping, check error log for skipped rows, verify record counts. For scheduled automations I verify the Last Run Status in the activity summary and set up monitoring so failed runs surface immediately rather than silently. At Cognizant I designed SQL data validation queries verifying 1M+ contact records post-migration using this workflow.

Module 1 — Case Management Architecture
Core of Service Cloud
📋
Case Lifecycle
New → Open → In Progress → Escalated → Pending Customer → Closed. Each stage has associated automation: Assignment Rules (who owns the case), Auto-Response Rules (acknowledgment email), Escalation Rules (time-based escalation), Validation Rules (data quality). QA must test every stage transition and the automation firing at each.
🔀
Assignment Rules
Route cases to queues or users based on criteria (Product, Region, Case Type, Priority). Evaluation order matters — first matching rule wins. QA: create test cases matching each rule criterion, verify routing. Test tie-breaker scenarios: what happens when no rule matches? (Goes to default queue — verify this is configured.)
⚡
Escalation Rules
Time-based — escalate case after N hours if not closed/updated. Can re-assign case, change priority, send email. QA: shortening escalation time in sandbox to test immediately is the standard approach. Test: case escalates correctly, escalation email sends to right person, case priority updates.
Module 2 — Omni-Channel Routing
Your strongest Service Cloud area
🌐
How Omni-Channel Works
Work items (Cases, Leads, Chats, Messaging Sessions) are pushed to available agents based on routing configuration. Two models: Queue-based routing (simplest — route to queue, first available agent gets item) and Skills-based routing (match work to agents with specific skills — language, product expertise). Presence Status controls whether agent can receive work.
⚖
Routing Configuration Objects
Routing Configuration: defines Priority (lower number = higher priority), Capacity (max work items per agent), Routing Model (Most Available, Least Active, External Routing). Service Channel: which object type is being routed. Queue: the pool work waits in. Agent Capacity: set per routing configuration. QA: test capacity limits — agent at max should NOT receive additional work items.
📊
Skills-Based Routing
Agents assigned Skills (e.g., "French Language", "Premium Support"). Work items tagged with required skills. Only agents with matching skills receive the work. QA test matrix: Agent A has Skill X only, Agent B has Skill Y only — verify Skill X work only goes to Agent A. Test: what if no agent with required skill is available?
🔴
Presence Status
Agents must be "Available" to receive work. Presence statuses: Available, Busy, Offline (customizable). When agent changes to Busy — no new work assigned. When status returns to Available — system automatically pushes next work item. QA: test mid-case status changes, verify in-progress work isn't reassigned when agent goes Busy.
Module 3 — Entitlements & SLAs
Compliance-critical feature
📜
Entitlement Architecture
Entitlement: a contractual right to support (e.g., "24/7 Phone Support"). Entitlement Process: defines the milestones and timeline. Milestone: a specific checkpoint in the case lifecycle (e.g., "First Response within 2 hours"). Each milestone has Success, Warning (approaching deadline), and Violation (deadline missed) actions.
🧪
How to Test Entitlements
Since milestones are time-based, test by: (1) Ask admin to temporarily set milestone time to 5 minutes in sandbox, (2) Create test case with entitlement, (3) Monitor milestone countdown on case detail page, (4) Verify Warning action fires before deadline, (5) Let deadline pass — verify Violation action fires (email alert, field update, task creation). Then restore to production timing.
Module 4 — Lightning Web Components (LWC) Testing
From your Qualitest experience
⚡
LWC Architecture Basics
LWC is Salesforce's modern component framework using web standards (Custom Elements, Shadow DOM). Components have: HTML template, JavaScript controller, optional CSS, metadata XML. @api properties for parent-to-child communication, custom events for child-to-parent. Wire adapters for declarative data fetching. QA focus: component behavior varies with data states and user permissions.
🧪
LWC Testing Strategy
Functional: test all data states (loading, loaded, empty, error). @api property testing: pass valid values, NULL, empty string, boundary values. Event testing: trigger user actions (click, input) and verify parent receives correct event data. CRUD operations: create/edit/delete records via component, verify in backend. Cross-browser: run BrowserStack tests on Chrome/Firefox/Safari (Shadow DOM behavior differs). Mobile: test on iOS Safari and Android Chrome for Service Console mobile compatibility.
🔐
Permission-Based Component Testing
LWC components often show/hide elements based on user permissions. Test as: Admin (full access), Standard User (limited), Partner/Community User (read-only). Components using @wire(getRecord) must be tested with users lacking Field-Level Security access to verify graceful degradation rather than JS errors.
What Modern Projects Do with Service Cloud
Modern Service Cloud implementations use Einstein Bots for AI-driven case deflection before human agents, Digital Engagement (WhatsApp, Apple Messages for Business) via Omni-Channel, Field Service Lightning for on-site work management, and Einstein Case Classification for auto-routing. QA teams are expected to test bot conversation flows, validate AI classification accuracy, and run Playwright-based regression against the Service Console UI. How do you test Omni-Channel routing end-to-end?
▼
I test Omni-Channel at four layers. Routing configuration: I verify Priority and Capacity settings are correct — create a test routing config in sandbox and validate its behavior before deploying to production. Queue assignment: I create test cases with different attribute combinations (region, product, case type) and verify each routes to the correct queue based on assignment rules. Capacity enforcement: I max out a test agent's capacity and verify the next work item goes to another available agent, not to the maxed-out one. Skills-based validation: I assign test agents specific skill sets and create work items requiring each skill — verify strict skill matching (a case requiring "French Language" doesn't route to an agent without that skill). I also test edge cases: what happens when no agent with required skill is available — does the case wait in queue or fall back to a default queue?

Explain the difference between Profiles, Permission Sets, and Sharing Rules and how you test each.
▼
Profiles define baseline access — object CRUD, field-level security, page layout, app visibility. Every user has exactly one profile. Permission Sets layer additional access on top of the profile without modifying it — they're additive only. Sharing Rules extend record-level access beyond the Org-Wide Default for specific groups. When testing: Profiles — I log in as a test user with that profile and manually verify CRUD operations on each object, check FLS by trying to view/edit restricted fields (should be hidden or read-only). Permission Sets — I assign/remove the PS to a test user mid-test and verify it grants exactly the intended additional access — then remove it and confirm access is revoked. Sharing Rules — I create records owned by Group A, switch to a Group B user and verify they can/can't access those records as configured. I also test direct URL record access — not just through list views — because some sharing issues only surface on direct navigation.
How do you test a Salesforce Flow?
▼
My Flow testing covers five areas. Trigger conditions: verify the Flow only fires when it should — test records that match trigger criteria AND records that don't. Entry conditions: if the Flow has entry filters, test records that pass and records that are excluded. Decision element paths: I create test records that route through each path of every Decision element, including NULL/else paths. Actions: verify every action (create record, update record, send email, call subflow) executes with correct data. Error handling: test what happens when a required field is NULL entering an action, or when a lookup returns no record — does the Flow fail gracefully or cause an unhandled fault? I run Flows in Debug mode in the Flow Builder first, then test via the actual trigger path in a sandbox before deploying.

Experience Cloud (Community Cloud)

Module 1 — What is Experience Cloud?
Conceptual foundation
🏗
Definition & Use Cases
Experience Cloud (formerly Community Cloud) lets organizations build branded digital experiences — portals, sites, apps — on top of Salesforce data. Authenticated users (customers, partners, employees) or unauthenticated guest users can interact with Salesforce records through these sites without accessing the full Salesforce UI.
📐
Key Templates
Customer Account Portal (self-service for customers — view cases, invoices), Partner Central (channel partner deal registration, leads), Help Center (public knowledge base), Microsites (marketing landing pages), Build Your Own (LWB/LWR custom templates). Each template has different default access and component sets.
👥
User Types
Internal Users: full Salesforce licenses. Community/Experience Cloud Users: cheaper external licenses (Customer Community, Customer Community Plus, Partner Community) with limited Salesforce access. Guest Users: completely unauthenticated — most restrictive access, assigned to the Guest User Profile. Each type has dramatically different access — QA must test as all three.
Module 2 — Access Control: The QA-Critical Layer
Most common source of production bugs
🔐
Org-Wide Defaults (OWD) in Experience Cloud
OWD defines the most restrictive access baseline. In Experience Cloud: if OWD for Cases is "Private", external users can only see their OWN cases. If "Public Read Only", all external users can read all cases. QA: always test record access from a standard external user, not just admin. The most common bug: dev tested as admin, which ignores sharing model entirely.
🗂
Guest User Security — Highly Regulated
Guest users are anonymous website visitors. They get the Guest User Profile — by default almost no access. Salesforce has tightened guest user security significantly (Salesforce Secure Guest User Access, 2021+). Guest users cannot access records owned by internal users without explicit sharing. QA: test all public-facing pages as a truly unauthenticated user (incognito browser) to verify no data leaks.
📋
Permission Sets for External Users
External users can receive Permission Sets just like internal users. Common pattern: base Customer Community license profile has minimal access; Permission Sets add access to specific objects, fields, and Apex classes. QA: test the "default" external user (profile only) and the "enhanced" user (profile + permission set) to verify each has exactly the right access — not too little, not too much.
Module 3 — Experience Cloud Testing Strategy
Your interview answer framework
👤
Persona-Based Testing Matrix
Create test users for each persona: Guest (unauthenticated), Standard External User (Customer Community profile), Enhanced External User (profile + permission sets), Internal Admin. Test each persona against every page, component, and data action. Your recruitment app example: Applicant (guest form submission) → Recruiter (internal review) → Admin (full access). Document what each persona CAN and CANNOT do.
🔗
URL-Based Access Testing
List view access ≠ direct URL access. A user might not see a record in a list (sharing rules exclude it) but could potentially access it via direct URL if OWD is too permissive. Always test: copy the direct record URL from an admin session, open in incognito/external user session, verify access is denied with appropriate error (not a blank page or JS error).
🔄
Flow Testing in Experience Cloud
Flows in Experience Cloud often run in the context of the external user — not in system context (unless "Run in System Context" is enabled). This means CRUD/FLS restrictions apply. QA: test Flows as the external user, not as admin. Common failure: Flow works in testing (admin), fails in production (external user) because the user lacks create permission on an object the Flow tries to create.
What Modern Projects Do with Experience Cloud
Modern Experience Cloud sites use LWR (Lightning Web Runtime) — the new high-performance rendering engine replacing Aura-based sites. They integrate with Salesforce CMS for content management, use Headless CMS APIs for custom frontends, and implement Salesforce Identity for SSO. QA teams now test headless API responses (not just UI), validate LWR site performance (Core Web Vitals matter), and run Playwright automation against the public-facing site in addition to the Salesforce admin UI.

Module 1 — Why Playwright? (vs Selenium/Cypress)
Interview context setter
⚡
Playwright Advantages
Auto-waiting: Playwright automatically waits for elements to be actionable before interacting — eliminates manual sleep() calls that cause flakiness. Cross-browser: supports Chromium, Firefox, and WebKit (Safari) natively. Network interception: intercept and mock API calls at the network layer. Parallel execution: built-in sharding and workers. Trace Viewer: record and replay test execution for debugging. No separate browser driver needed (unlike Selenium WebDriver).
🔄
vs Selenium
Selenium: older, requires WebDriver setup, more verbose API, no built-in auto-wait (requires explicit waits), browser driver version management headaches. Playwright: modern API, auto-wait built in, better debugging tools (Trace Viewer, Inspector), faster execution. When Selenium is still used: legacy codebases, teams with large existing Selenium investment, specific Selenium Grid infrastructure already in place.
🌀
vs Cypress
Cypress: excellent for JavaScript-heavy SPAs, runs inside the browser (different architecture), great developer experience. Limitations: Chrome/Electron only (limited WebKit), can't test multiple tabs/windows, challenging for non-JavaScript stacks. Playwright: cross-browser including Safari, supports multi-tab and multi-origin scenarios, better for enterprise apps like Salesforce.
Module 2 — Core Playwright Architecture
Technical depth for senior roles
🌐
Browser Contexts
A Browser Context is an isolated browser environment — separate cookies, localStorage, session state. Like an incognito window. Use contexts to: run tests in parallel with different user sessions, test multi-user scenarios (agent + customer simultaneously), test logged-in and logged-out states in the same test. Create with browser.newContext(). In Playwright, each test automatically gets its own context.
🎯
Locator Strategies — Best to Worst
1st choice: page.getByRole('button', {name:'Submit'}) — semantic, resilient. 2nd: page.getByLabel('Email') — accessibility-based. 3rd: page.getByText('Continue') — visible text. 4th: page.getByTestId('submit-btn') — data-testid attribute. Avoid: CSS selectors with dynamic class names, XPath, index-based selectors. In Sentinel your self-healing layer prioritizes semantic locators and falls back to structural analysis when primary locators break.
🕐
Auto-Wait & Assertions
Playwright auto-waits for: element to be visible, enabled, stable (not animating), and attached to DOM before acting. Use expect(locator).toBeVisible() for assertions — this retries internally until the default timeout (5s). Use page.waitForResponse(/api\/endpoint/) to wait for specific network responses before proceeding. Never use page.waitForTimeout(2000) — fixed delays create flaky, slow tests.
🔌
Network Interception & API Mocking
page.route('/api/data', route => route.fulfill({json: mockData})) intercepts network requests and returns mock data. Use for: testing UI in different data states without a live backend, simulating API errors (return 500), testing loading states (add delay). page.waitForResponse() waits for real API calls and lets you assert on the response.
Module 3 — Page Object Model (POM)
Architecture pattern — critical for senior roles
🏗
POM Structure
Each page/major component = one class. Class encapsulates all locators as private properties and all interactions as public methods. Tests never call page.locator() directly. Structure: LoginPage, CasesListPage, CaseDetailPage, etc. A BasePage class holds shared utilities (navigation, common assertions). This is what Sentinel's Test Generation Agent outputs — compliant POM-structured TypeScript files.
💡
POM Code Pattern
class LoginPage {
  private emailInput = this.page.getByLabel('Email');
  private passwordInput = this.page.getByLabel('Password');
  private submitBtn = this.page.getByRole('button',{name:'Log in'});

  constructor(private page: Page) {}

  async login(email: string, password: string) {
    await this.emailInput.fill(email);
    await this.passwordInput.fill(password);
    await this.submitBtn.click();
  }
}
🔧
Fixtures for Shared State
Playwright fixtures extend the test object with reusable setup/teardown. Use fixtures for: authenticated page objects (auto-login before every test), seeded test data (create test records, clean up after), shared browser context config. Fixture example: test.extend({ loggedInPage: async ({page}, use) => { await page.goto('/login'); await page.fill... await use(page); } })

How do you handle flaky tests in Playwright?
▼
I approach flaky tests at three layers. Prevention first: use Playwright's semantic locators (getByRole, getByLabel) instead of brittle CSS selectors — they survive UI refactoring. Use expect() assertions with built-in retry instead of fixed delays. Use waitForResponse() to wait for real API calls rather than arbitrary timeouts. Detection: track flaky test rate in CI — a test that sometimes passes and sometimes fails is a bug in the test, not just bad luck. Diagnosis: when a test is flaky, run it in headed mode with --debug to observe the failure, then analyze the Playwright Trace Viewer which records every action, network call, and screenshot. Common flakiness causes: race conditions between UI rendering and assertion, network timing variance, test data conflicts between parallel workers. For parallel execution I ensure test data is isolated per worker using unique identifiers. In Sentinel's self-healing layer I also track locator failure patterns to distinguish true application bugs from UI change-induced locator failures.
Walk me through your Page Object Model approach.
▼
POM is a design pattern that separates test logic from page interaction details. Each page or major component in the application gets its own class. The class encapsulates all locators as private properties using semantic Playwright locators, and all interactions as public methods. Tests import these page classes via fixtures and call methods like loginPage.login(credentials) — they never call page.locator() directly. This means when the UI changes, I update only the Page Object class, not every test that uses it. I also maintain a BasePage class for shared utilities — navigation helpers, common wait assertions, screenshot capture on failure. This structure is exactly what Sentinel's Test Generation Agent produces — compliant, maintainable POM TypeScript files. For large applications I organize page objects in a directory structure mirroring the app's navigation (auth/, cases/, omni-channel/), making them discoverable for the whole team.

Module 1 — CI/CD Fundamentals
How to explain it in any interview
🔄
What is CI?
Continuous Integration: every time a developer pushes code or opens a PR, an automated pipeline runs — builds the application and executes the test suite. Goal: catch bugs immediately on every change, before they compound. Key principle: the main branch is always in a deployable state. If CI fails, fixing it is the team's top priority.
🚀
What is CD?
Continuous Delivery: when CI passes, code is automatically deployed to a staging/pre-production environment. Teams can then deploy to production with one click. Continuous Deployment (stronger): code is automatically deployed to production after all tests pass — no manual gate. QA role: ensure automated test coverage is reliable enough to trust CD without manual sign-off for routine changes.
🎯
QA's Role in CI/CD
The automated regression suite is the quality gate. QA engineers: write and maintain the test suite that runs in CI, set the threshold for pass/fail (e.g., 95% pass rate required), define which tests block a merge (smoke) vs which are informational, monitor flaky test rate, and integrate test results into team dashboards (Jira, Slack). "I treat the CI pipeline as the last defence before code reaches users — my tests are the gate."
Module 2 — GitHub Actions Deep Dive
Your Horizontal Digital experience
📄
Workflow YAML Structure
Workflows live in .github/workflows/ as YAML files. Key elements: on (triggers — push, pull_request, schedule, workflow_dispatch), jobs (parallel or sequential execution units), steps (individual commands or actions). Each job runs on a fresh virtual machine (runs-on: ubuntu-latest).
⚙
Playwright in GitHub Actions
name: Playwright Tests
on:
  pull_request:
    branches: [main]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: {node-version: '20'}
      - run: npm ci
      - run: npx playwright install --with-deps
      - run: npx playwright test
      - uses: actions/upload-artifact@v4
        if: failure()
        with:
          name: playwright-report
          path: playwright-report/

Matrix Strategy for Cross-Browser
Run same tests across multiple configurations in parallel: strategy: matrix: browser: [chromium, firefox, webkit]. Each combination runs as a separate job simultaneously. Use for cross-browser testing without sequential waiting. GitHub Actions provides 20 parallel jobs on free plan, more on paid — this enables fast parallel test execution.

Module 4 — Shift-Left Testing & DevOps Culture
Career positioning topic
⬅
What is Shift-Left?
Move testing earlier in the SDLC. Traditional waterfall: Design → Code → Test → Deploy (testing is last). Shift-left: QA involved in requirements review, three-amigos sessions (QA + Dev + BA), writing acceptance criteria, automated tests written alongside code, tests running on every commit. Result: bugs caught earlier when they're cheapest to fix — fixing a bug in requirements costs 1x; fixing in production costs 100x.
🧪
Test Pyramid in CI/CD
Unit Tests (bottom — most, fastest, cheapest): developer writes, runs on every commit, <30s. Integration Tests (middle): test service interactions, run on PR, ~2-5 min. E2E/UI Tests (top — fewest, slowest, most expensive): Playwright tests, run on PR merge to main, ~10-20 min. QA owns the E2E layer and integration layer. The pyramid shape matters: if E2E tests outnumber unit tests, the pipeline is slow and expensive to maintain.
📈
Release Risk Scoring (Your Sentinel Feature)
One of Sentinel's features for mid-market engineering teams. Analyzes: which files changed in a PR, historical failure rate of tests covering those files, test coverage gaps in changed areas. Outputs a risk score (Low/Medium/High) that helps teams decide how extensive their pre-release testing should be. This is a real DevOps concept called "risk-based testing" — mentioning this in an interview shows senior-level thinking.

Modern DevOps teams implement Trunk-Based Development (all developers commit to main, feature flags control releases). GitOps for infrastructure (Terraform, Kubernetes manifests in git, automatically applied). Security scanning in CI (SAST, dependency vulnerability checks — Snyk, Dependabot). Performance testing in CI (k6, Artillery for load tests on PRs).
