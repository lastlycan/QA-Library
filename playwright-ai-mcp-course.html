What is Playwright & Why It Matters
Intro · 5 min

Overview

Playwright is Microsoft's open-source browser automation library. It supports Chromium, Firefox, and WebKit from a single API — and it's the fastest-growing automation tool in the industry.

Key Points

  • Cross-browser: Chrome, Firefox, Safari (WebKit) with a single codebase
  • Auto-waits — no more sleep() or flaky timing hacks
  • Built-in test runner, tracing, screenshots, and video recording
  • Supports TypeScript, JavaScript, Python, Java, .NET
  • Native support for parallel execution and sharding

Writing Your First Test
Hands-On · 10 min
NEW

Basic Test Structure

Every Playwright test uses test() and receives a page fixture. Here's a complete test you can run immediately:

import { test, expect } from '@playwright/test';

test('homepage has correct title', async ({ page }) => {
  // Navigate to a URL
  await page.goto('https://playwright.dev');

  // Assert page title
  await expect(page).toHaveTitle(/Playwright/);

  // Click the "Get Started" link
  await page.getByRole('link', { name: 'Get started' }).click();

  // Verify navigation happened
  await expect(page).toHaveURL(/.*intro/);
});
# Run your tests
npx playwright test

# Run with UI mode (interactive)
npx playwright test --ui

# Run headed (see the browser)
npx playwright test --headed

Key Points

  • Prefer getByRole() over CSS selectors — it's more resilient
  • Always await Playwright actions — they're all promises
  • Use expect() assertions with auto-retry built in

Locators & Selectors Deep Dive
Core Skill · 12 min
NEW

The Locator Hierarchy (Best → Worst)

Playwright's locator strategy is key to writing stable tests. Follow this priority order:

// ✅ BEST — role-based (semantic, accessible)
page.getByRole('button', { name: 'Login' })
page.getByRole('textbox', { name: 'Email' })

// ✅ GOOD — text-based
page.getByText('Welcome back!')
page.getByLabel('Password')
page.getByPlaceholder('Enter your email')

// ✅ GOOD — test IDs (recommended for complex apps)
page.getByTestId('submit-btn') // data-testid="submit-btn"

// ⚠️  AVOID — brittle selectors
page.locator('.btn-primary') // CSS class can change
page.locator('#login-btn') // IDs can change
page.locator('//button[1]') // XPath is fragile

Chaining & Filtering

// Filter by child element
page.getByRole('listitem')
    .filter({ hasText: 'Product 1' })
    .getByRole('button', { name: 'Add to cart' })

// Count items
await expect(page.getByRole('listitem')).toHaveCount(3);

Page Object Model (POM)
Architecture · 15 min
NEW

Why POM?

The Page Object Model separates test logic from page interaction details. When the UI changes, you update one place — not every test.

// pages/LoginPage.ts
import { Page, Locator } from '@playwright/test';

export class LoginPage {
  private emailInput: Locator;
  private passwordInput: Locator;
  private submitBtn: Locator;

  constructor(private page: Page) {
    this.emailInput = page.getByLabel('Email');
    this.passwordInput = page.getByLabel('Password');
    this.submitBtn = page.getByRole('button', { name: 'Sign in' });
  }

  async goto() {
    await this.page.goto('/login');
  }

  async login(email: string, password: string) {
    await this.emailInput.fill(email);
    await this.passwordInput.fill(password);
    await this.submitBtn.click();
  }
}

// tests/login.spec.ts
test('successful login', async ({ page }) => {
  const loginPage = new LoginPage(page);
  await loginPage.goto();
  await loginPage.login('user@test.com', 'secret123');
  await expect(page).toHaveURL('/dashboard');
});

import fs from 'fs';

test('AI visual validation', async ({ page }) => {

      role: 'user',
      content: [
        {
          type: 'image',
          source: {
            type: 'base64',
            media_type: 'image/png',
            data: screenshot.toString('base64')
          }
        },
        {
          type: 'text',

        }

MCP (Model Context Protocol) is an open standard by Anthropic that lets AI models communicate with external tools, data sources, and services through a unified interface. Think of it as USB-C for AI integrations.

                                      │  - Databases    │
                                      │  - APIs         │
                                      │  - File system  │
                                      └─────────────────┘

Fixtures & Custom Test Context
Architecture · 15 min
NEW

What Are Fixtures?

Fixtures are Playwright's dependency injection system. They let you set up shared context (logged-in user, seeded DB, pre-navigated page) once and reuse it across tests without copy-pasting setup code.

Built-in Fixtures

// Playwright provides these out of the box:
test('example', async ({ page, context, browser, request }) => {
  // page     → a fresh Page for each test
  // context  → a BrowserContext (cookies, storage)
  // browser  → the Browser instance
  // request  → APIRequestContext for HTTP calls
});

Custom Fixtures — Logged-In User

// fixtures/auth.ts
import { test as base, expect } from '@playwright/test';
import { LoginPage } from '../pages/LoginPage';

type AuthFixtures = {
  authenticatedPage: Page;
};

export const test = base.extend<AuthFixtures>({
  authenticatedPage: async ({ page }, use) => {
    // Setup: log in before the test
    const login = new LoginPage(page);
    await login.goto();
    await login.login('admin@test.com', 'pass123');
    await expect(page).toHaveURL('/dashboard');

    // Hand the page to the test
    await use(page);

    // Teardown (optional): clean up after test
    await page.goto('/logout');
  }
});

// tests/dashboard.spec.ts
import { test } from '../fixtures/auth';

test('dashboard shows widgets', async ({ authenticatedPage }) => {
  // Already logged in! No setup needed.
  await expect(authenticatedPage.getByRole('heading', { name: 'Dashboard' })).toBeVisible();
});

Faster Login with Storage State

// global-setup.ts — run once before all tests
import { chromium } from '@playwright/test';

export default async function globalSetup() {
  const browser = await chromium.launch();
  const page = await browser.newPage();
  await page.goto('http://localhost:3000/login');
  await page.getByLabel('Email').fill('admin@test.com');
  await page.getByLabel('Password').fill('secret');
  await page.getByRole('button', { name: 'Login' }).click();

  // Save cookies + localStorage — reused by all tests!
  await page.context().storageState({ path: 'auth.json' });
  await browser.close();
}

// playwright.config.ts
export default {
  globalSetup: './global-setup.ts',
  use: { storageState: 'auth.json' } // inject saved session
};
  • Storage state login is 10× faster than UI login per test
  • Fixtures scope: test (default), worker, or session
  • Worker-scoped fixtures are shared across tests in the same worker process

API Mocking & Network Interception
Advanced · 15 min
NEW

Why Mock APIs?

Network interception lets you test edge cases (errors, timeouts, empty states) that are hard to reproduce with a real backend, and speeds up tests by 3-5× by eliminating real HTTP calls.

Intercept & Mock a Response

test('shows empty state when no products', async ({ page }) => {
  // Intercept before navigating
  await page.route('**/api/products', route => {
    route.fulfill({
      status: 200,
      contentType: 'application/json',
      body: JSON.stringify([]) // empty list
    });
  });

  await page.goto('/products');
  await expect(page.getByText('No products found')).toBeVisible();
});

test('handles API error gracefully', async ({ page }) => {
  await page.route('**/api/products', route => {
    route.fulfill({ status: 500, body: 'Internal Server Error' });
  });

  await page.goto('/products');
  await expect(page.getByRole('alert')).toContainText('Something went wrong');
});

Modify (Not Replace) Real Responses

await page.route('**/api/user', async route => {
  // Let the real request go through...
  const response = await route.fetch();
  const json = await response.json();

  // ...then modify the response
  json.role = 'admin'; // pretend the user is an admin
  await route.fulfill({ json });
});

Spy on Requests (No Interception)

// Assert an API call was made with correct payload
const requestPromise = page.waitForRequest('**/api/orders');
await page.getByRole('button', { name: 'Place Order' }).click();

const request = await requestPromise;
expect(request.method()).toBe('POST');
expect(request.postDataJSON()).toMatchObject({ items: [{ id: 42 }] });

Soft Assertions — Don't Stop on First Failure

test('full page audit', async ({ page }) => {
  await page.goto('/dashboard');

  // soft assertions — test continues even if these fail
  await expect.soft(page.getByRole('heading')).toBeVisible();
  await expect.soft(page.getByText('Revenue')).toBeVisible();
  await expect.soft(page.getByRole('table')).toBeVisible();

  // hard assertion — fails here if any soft ones failed
  expect(test.info().errors).toHaveLength(0);
});
  • Use trace: 'on-first-retry' in CI to avoid trace overhead on green runs
  • Playwright's HTML report shows test timeline, screenshots, and videos in one place
  • Soft assertions are great for visual audits — collect all failures at once

GitHub Actions — Full CI Pipeline
DevOps · 15 min
NEW

Complete GitHub Actions Workflow

This workflow runs Playwright tests on every push and PR, uploads reports as artifacts, and posts results as a PR comment.

# .github/workflows/playwright.yml
name: Playwright Tests

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    timeout-minutes: 30

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Install Playwright Browsers
        run: npx playwright install --with-deps

      # Optional: start your app locally
      - name: Start app
        run: npm run start &
        env:
          DATABASE_URL: ${{ secrets.DATABASE_URL }}

      - name: Run Playwright Tests
        run: npx playwright test
        env:
          BASE_URL: http://localhost:3000
          CI: true

      # Always upload report, even on failure
      - uses: actions/upload-artifact@v4
        if: always()
        with:
          name: playwright-report
          path: playwright-report/
          retention-days: 30

Sharding for Parallelism

# Run 4 shards in parallel — 4× faster!
strategy:
  matrix:
    shard: [1, 2, 3, 4]

steps:
  - name: Run tests (shard ${{ matrix.shard }}/4)
    run: npx playwright test --shard=${{ matrix.shard }}/4

  - uses: actions/upload-artifact@v4
    if: always()
    with:
      name: blob-report-${{ matrix.shard }}
      path: blob-report/

# Merge reports from all shards
merge-reports:
  needs: [test]
  runs-on: ubuntu-latest
  steps:
    - run: npx playwright merge-reports --reporter html ./all-blob-reports
    - uses: actions/upload-artifact@v4
      with:
        name: playwright-report
        path: playwright-report/
  • Always use --with-deps when installing browsers in CI — it installs system libs
  • Cache ~/.cache/ms-playwright to avoid re-downloading browsers every run
  • Sharding cuts CI time proportionally — 4 shards = ~4× speedup
  • Set CI: true env var — Playwright automatically retries flaky tests in CI

playwright.config.ts — Master Configuration
Config · 10 min

Production-Ready Config

// playwright.config.ts
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  testDir: './tests',
  fullyParallel: true,
  forbidOnly: !!process.env.CI,   // fail if test.only committed
  retries: process.env.CI ? 2 : 0, // retry twice in CI only
  workers: process.env.CI ? 4 : undefined,
  reporter: [
    ['html'],
    ['junit', { outputFile: 'results.xml' }], // for CI integrations
  ],

  use: {
    baseURL: process.env.BASE_URL ?? 'http://localhost:3000',
    trace: 'on-first-retry',
    screenshot: 'only-on-failure',
    video: 'retain-on-failure',
    actionTimeout: 10_000,    // 10s per action
    navigationTimeout: 30_000, // 30s per navigation
  },

  projects: [
    // Setup project (runs first)
    { name: 'setup', testMatch: '**/auth.setup.ts' },

    // Test projects
    {
      name: 'chromium',
      use: { ...devices['Desktop Chrome'] },
      dependencies: ['setup'],
    },
    {
      name: 'firefox',
      use: { ...devices['Desktop Firefox'] },
      dependencies: ['setup'],
    },
    {
      name: 'mobile-safari',
      use: { ...devices['iPhone 14'] },
      dependencies: ['setup'],
    },
  ],

  // Start dev server before tests run
  webServer: {
    command: 'npm run dev',
    url: 'http://localhost:3000',
    reuseExistingServer: !process.env.CI,
  },
});
  • forbidOnly prevents accidentally merging test.only to main
  • webServer auto-starts your app and waits for it to be ready
  • devices presets configure viewport, user agent, and touch correctly
  • Use dependencies to ensure auth setup runs before test projects
