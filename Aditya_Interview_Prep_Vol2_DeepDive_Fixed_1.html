SFMC — Advanced Q&A

12 advanced questions interviewers ask senior SFMC QA engineers
What's the difference between Triggered Sends and Journey Builder for transactional emails?
▼
Triggered Sends (via Email Studio) are optimized for immediate 1:1 transactional sends — order confirmations, password resets, OTPs. They fire via API with minimal latency and are designed for single-email transactions without orchestration. Journey Builder is better when you need multi-step workflows — send an email, wait 2 days, check if clicked, send a follow-up. Key differences: Triggered Sends have lower latency and simpler setup; Journey Builder supports complex branching and multi-channel orchestration but has higher processing overhead. For pure transactional high-volume sends (millions of order confirmations per day), Triggered Send is the right architecture. For nurture, lifecycle, or multi-step campaigns, Journey Builder wins. When testing Triggered Sends: validate API payload structure matches the Triggered Send Definition exactly, verify AMPscript renders correctly with payload data (not DE lookup), and test send speed under expected load volume.
Interview angle
Mentioning this distinction shows architectural understanding, not just feature knowledge — it signals senior-level thinking.
How do you test suppression lists and opt-out compliance in SFMC?
▼
Suppression testing is highest-risk in SFMC QA — sending to someone who opted out is a CAN-SPAM/GDPR violation. My testing covers three layers. Global suppression: I add a test email to All Subscribers as Unsubscribed, then trigger a send including that address — verify the contact appears in _Sent with Suppressed status, not Sent. List-level unsubscribe: I unsubscribe a test contact from Publication List A, then send from Publication List B — verify they receive List B (list-level unsubscribe is more granular than global). Suppression DE: many orgs use a custom suppression DE in Automation Studio SQL to exclude contacts before send. I test that SQL correctly excludes all suppressed contacts and that the suppression DE is refreshed before every send. Finally, I validate the preference center — clicking "Unsubscribe" in a real email updates subscriber status within SFMC within the expected processing time.
Explain Marketing Cloud Connect — what can go wrong from a QA perspective?
▼
Marketing Cloud Connect bridges Salesforce CRM and SFMC. The QA risks: (1) Subscriber Key mismatch — if Subscriber Key in SFMC doesn't match the Contact ID in CRM, sends fail silently. I test by cross-referencing Subscriber Keys between Synchronized DE and _Subscribers data view on a random sample. (2) Sync delays — MCC syncs every 15 minutes by default. I test time-sensitive journeys by verifying that a CRM Contact change propagates to SFMC within the expected sync window before the next scheduled send. (3) Field mapping errors — if a CRM field is renamed, the Synchronized DE mapping can break silently. I test by making a test field change in sandbox CRM and verifying the sync doesn't error. (4) Deleted record handling — I test what happens in SFMC when a CRM Contact is deleted: subscriber should remain active in SFMC for historical tracking, not be removed.
Your proof point
At Cognizant: validated CRM-to-SFMC sync integrity across 1M+ contact records post-migration before loyalty campaign go-live.
A marketing send went to 50,000 wrong contacts. Walk me through your investigation and prevention.
▼
Immediate investigation: pull _Sent data view to identify all subscribers who received the send, cross-reference against the intended audience DE to understand scope. Trace back to the entry source — was this a Journey or direct send? Was a wrong DE used? Did a Decision Split path misbehave? Root cause analysis — most wrong-audience sends come from: (1) using the wrong DE as entry source (naming confusion, wrong environment), (2) SQL query bug in Automation Studio producing incorrect segment, (3) Decision Split condition error routing everyone to the same path, or (4) no pre-send audience review under sprint pressure. Prevention I'd implement: mandatory pre-send QA checklist with audience SIZE validation (if expected 5,000 but DE shows 50,000, stop and investigate), test contact seeding and content preview before every send, SQL query peer review for Automation Studio, staging dry-run with count verification before production execution, and an alert when send audience is more than 20% larger than the previous comparable send.
Show process maturity
Proposing preventive controls — not just fixing the immediate issue — demonstrates senior-level thinking that interviewers look for.
How do you test multi-business-unit (BU) setups in SFMC?
▼
Multi-BU setups have a parent Enterprise account with child Business Units for each brand or region. Subscribers are managed at Enterprise level (shared All Subscribers list) while sends happen at BU level. QA considerations: Subscriber isolation — verify DEs and subscribers in BU A are not accessible from BU B unless shared data extensions are explicitly configured. From address management — each BU has its own verified From Addresses; test that emails from BU A use BU A's domain not BU B's. Shared templates — if Content Builder assets are shared across BUs, test that a template change in the parent doesn't break child BU email rendering. Cross-BU journey entry — if a journey in BU A triggers sends from BU B, verify the subscriber exists in the correct publication list for both BUs. This scenario came up at Cognizant with a global restaurant chain that had regional BUs — I had to test that UK subscribers received emails from the UK BU domain and US subscribers from the US BU domain, with correct personalization for each region.
How does SFMC handle bounce management and what do you specifically test?
▼
SFMC automatically manages bounces and updates subscriber status. Hard bounces (permanent — address doesn't exist) cause SFMC to set subscriber status to "Bounced" — excluded from future sends. Soft bounces (temporary — mailbox full, server timeout) are tracked but status doesn't change immediately; after a configured threshold of soft bounces, the subscriber is treated as a hard bounce. For QA testing: I validate the _Bounce data view after a test send to verify bounce events are captured with correct BounceType codes (HardBounce vs SoftBounce). I verify a contact with Bounced status in All Subscribers is excluded from subsequent sends (check _Sent status — should show Excluded, not Sent). I test the bounce notification delivery to the designated monitoring inbox. I also test bounce processing timing — bounce status updates aren't always immediate, so I verify within the expected processing window rather than immediately after send.
What is Einstein in SFMC and how do you test non-deterministic AI features?
▼
Einstein in SFMC includes Send Time Optimization (STO — AI-driven per-individual send timing), Einstein Content Selection (dynamic content based on predicted engagement), Einstein Recommendations (product/content recommendations via AMPscript), and Engagement Scoring (subscriber engagement likelihood scores). AI features are non-deterministic — you can't test for exact output. Instead I validate the integration and behavior boundaries: Einstein STO enabled on a journey send activity — verify individual send times vary across test contacts (not all at identical time). For Einstein Recommendations: verify the REST API call returns recommendations in the expected JSON schema, and that AMPscript renders recommended products correctly — testing with contacts who have and don't have sufficient engagement history (fallback content must render correctly for new subscribers with no history). For STO: verify the feature degrades gracefully when there's insufficient send history — falls back to a configured default send time, not no send at all.
How do you test email rendering across clients?
▼
Email rendering is notoriously inconsistent — HTML/CSS that works in Gmail breaks in Outlook, which uses Microsoft Word's rendering engine (not a real browser engine). My approach: I use Litmus or Email on Acid integrated with SFMC for automated rendering previews across 90+ email clients. Before every template launch I specifically check: Outlook 2019/365 (most restrictive — no CSS Grid, no Flexbox, limited modern CSS), Gmail App on Android (different from Gmail desktop), Apple Mail on iOS (supports most features but has unique quirks), and dark mode rendering (content must remain readable when OS inverts colors). I also test AMPscript's impact on rendering — server-side code errors can break the HTML structure entirely. And I validate images: correct dimensions, alt texts showing when images are blocked (common in corporate clients), fallback fonts when specified font isn't available, and mobile responsiveness. I document rendering issues with screenshots and categorize as critical (unreadable content) vs cosmetic (minor layout shift).

10 advanced questions on Service Cloud depth topics
How do you test Einstein Bots / AI Chat in Service Cloud?
▼
Einstein Bot testing is unique because you're testing both conversation logic and AI intent classification. Intent coverage: I create a test matrix of utterances for every configured intent ("I need help with my order", "How do I track my package") and verify the bot classifies them correctly into the right dialog path. I also test edge cases — typos, ambiguous phrasing, multi-intent utterances. Fallback testing: I deliberately use phrases that don't match any intent to verify the bot escalates to a human agent gracefully rather than looping or giving a confusing response. Variable extraction: for bots that extract entities (order number, email) from user input, I test with valid formats, invalid formats, and missing information. Escalation paths: test every escalation trigger — explicit "speak to agent" request, confusion after N failed intent matches, after-hours escalation. Handoff quality: does the agent receive the full chat transcript? Are extracted variables correctly passed to the case or agent workspace? Finally, I test across web chat and mobile chat surfaces since rendering and bot behavior can differ.
What's your approach to regression testing for a Salesforce major release? (3 per year)
▼
Salesforce's three major releases can impact customizations, LWC behavior, Flow functionality, and API compatibility. My pre-release regression strategy: six weeks before the release, I run the full automated Playwright regression suite against the pre-release sandbox (Salesforce offers pre-release orgs — request access at help.salesforce.com). I pay particular attention to "Retirement" notices in the release notes — if a feature we use is being retired, I prioritize those test cases immediately. I activate "Critical Updates" in sandbox early — these are changes that become mandatory at the release date; testing them early means the production release isn't a surprise. For LWC-heavy apps: I specifically run cross-browser regression since Lightning updates can change Shadow DOM behavior. I also verify API version compatibility — are our Playwright API test calls still valid with the new API version? I produce a release regression report documenting what was tested, what passed, what failed, and any required workarounds — this becomes the go/no-go input for the production release decision.

Describe how you'd build a full test plan for a new Service Cloud implementation.
▼
I structure Service Cloud test plans in six phases. Phase 1 — Configuration smoke test: verify all setup is complete — Omni-Channel routing configurations, queue memberships, entitlement processes, assignment rules, user profiles/permission sets. This is setup verification before business logic testing. Phase 2 — Unit testing automations: each assignment rule entry, each escalation rule trigger, each Flow path tested in isolation with targeted test records. Phase 3 — Integration testing: end-to-end case lifecycle — case creation triggers correct assignment rule, Omni-Channel pushes to correct agent, entitlement milestone clock starts, escalation fires on schedule, case closure triggers CSAT survey. Phase 4 — Persona UAT: test as Tier 1 agent, Tier 2 agent, supervisor, and customer (if Experience Cloud portal exists). Phase 5 — Volume testing: simulate peak concurrent case volume — does Omni-Channel routing hold up under load? Phase 6 — Regression suite: automate the 20 highest-risk scenarios in Playwright for ongoing CI regression. Every test case maps to a business requirement in the test plan — making UAT sign-off structured not ad-hoc.

How do you test bulk case operations like mass reassignment or mass status update?
▼
Bulk operations interact with Salesforce governor limits, triggers, and automation differently than single-record operations. For mass case reassignment: I test with small batches first (10 cases) to verify expected behavior — correct owner assignment, Omni-Channel work item routing update, notification emails sent. Then I scale to larger batches (100, 1000) to check governor limit issues (SOQL query limits in Flows/triggers, DML row limits). Specific checks: does the assignment rule re-evaluate on mass update? (Bulk updates can sometimes bypass certain automation.) Does the Omni-Channel work item correctly reroute when a case owner changes? For mass status updates (e.g., closing all cases in a queue): verify escalation rules stop evaluating for closed cases, entitlement milestones are marked complete rather than violated, and case close automation (CSAT trigger) fires for each closed case — not just the first. I also test partial failure rollback: if a mass update hits a validation rule on record 500 of 1000, does Salesforce roll back all 1000 or only the 500 that succeeded? (It rolls back all — atomic operation.)
How do you manage Salesforce sandbox strategy for QA — which sandboxes, for what purpose?
▼
Sandbox types matter significantly for QA. Developer sandbox: limited storage, no production data — good for unit testing individual configurations and automation. Developer Pro: more storage, still no production data. Partial Copy: random 10,000-record sample from production — good for realistic data testing and integration testing with representative data volumes. Full Copy: exact production copy — gold standard for UAT, release regression, and performance testing; expensive but essential for high-stakes implementations. My sandbox strategy: Developers work in Developer sandboxes. Integration testing happens in a shared Developer Pro with controlled test data. UAT happens in Partial Copy (realistic data without full production volume). Release regression and performance testing in Full Copy before major go-lives. For SFMC: I maintain dedicated test Data Extensions populated with internal email addresses only — never real customer data in any sandbox. At Horizontal Digital's migration project, I used a Partial Copy sandbox for volume testing before migrating 1M+ contacts to production.

Experience Cloud — Advanced Q&A

8 advanced questions on Experience Cloud depth
How do you test Experience Cloud performance and Core Web Vitals?
▼
Performance testing for Experience Cloud is important because customer-facing portals directly impact conversion and satisfaction — and Google factors Core Web Vitals into SEO ranking. My approach: I use Playwright's built-in performance APIs to measure page load timing. I run Lighthouse audits via Chrome DevTools Protocol in Playwright to capture Core Web Vitals — Largest Contentful Paint (LCP, target <2.5s), First Input Delay (FID, target <100ms), and Cumulative Layout Shift (CLS, target <0.1). For Experience Cloud specifically, I test as unauthenticated guest user (most common for public sites) and authenticated external user — they have different caching behavior and data payloads. I look for: LWC components triggering multiple serial API calls during page load (waterfall delays), large unoptimized images (the #1 LCP killer), components loading synchronously when they should be lazy-loaded, and layout shift caused by content loading after initial render. I run performance tests in pre-release sandbox after every sprint to catch performance regressions before go-live, not just in final testing.
How do you test SSO integration with Experience Cloud?
▼
SSO for Experience Cloud (typically SAML 2.0 or OAuth 2.0) bridges the Identity Provider (Okta, Azure AD) and Salesforce as Service Provider. My SSO testing covers: Happy path — navigate to site, click Login, redirect to IdP login page, authenticate with valid credentials, redirect back as the correct user with correct profile. Identity assertion validation — the SAML assertion or OAuth token contains an email/username that maps to an existing Salesforce External User record; test what happens if the assertion email doesn't exist in Salesforce (should show a clear error, not a blank page or cryptic Salesforce error). Session timeout — when IdP session expires, user should be prompted to re-authenticate rather than encountering a cryptic error. Logout — verify logout terminates both Salesforce and IdP session (where configured), not just Salesforce. Error handling — test invalid credentials, locked IdP account, network failures during the redirect flow. I always test SSO in a non-admin browser session — admin sessions bypass SSO configurations and give false confidence.
How do you test accessibility (a11y) on an Experience Cloud site?
▼
Accessibility testing for Experience Cloud matters both legally (WCAG 2.1 AA compliance) and for reach. Automated: I integrate @axe-core/playwright into the test suite — await checkA11y(page) after page load flags ARIA issues, color contrast failures, missing alt text, and form label problems. This runs in CI on every PR so accessibility regressions are caught immediately. Manual: Keyboard navigation — tab through all interactive elements in logical order, verify focus states have visible outlines, verify modal closes with ESC, verify skip-to-content link works. Screen reader testing: I test with NVDA (Windows) or VoiceOver (macOS/iOS) — LWC Shadow DOM can cause screen readers to miss dynamic content loaded by wire adapters. Specific Experience Cloud checks: form validation errors must be associated with their fields programmatically via aria-describedby (not just visually red), dynamic content updates (e.g., chat messages, case status changes) must announce via aria-live regions, image carousels need pause buttons. I document accessibility issues with WCAG success criteria references to help developers understand the fix required.
Describe testing a headless Experience Cloud implementation.
▼
Headless Experience Cloud uses Salesforce as the backend only — serving content and data via APIs to a custom frontend (React, Next.js, Vue). This changes QA fundamentally. Instead of testing the Experience Builder UI, I test the Salesforce APIs the frontend consumes: CMS Delivery API (content), Salesforce Experience API (record data), Salesforce Identity API (authentication). API contract testing: I document expected API response schemas (field names, types, structures) and write Playwright API tests that validate these contracts. Any API change that breaks the frontend contract is a critical regression. Authentication flow: the custom frontend manages OAuth token lifecycle — I test token acquisition, silent refresh, and expiry handling. How do you architect a Playwright test suite for a large enterprise application?
▼
For an enterprise app (50+ pages, multiple user roles, complex workflows) I follow a layered architecture. Directory structure: /tests (spec files by feature — auth/, cases/, routing/), /pages (POM classes mirroring app navigation), /fixtures (test.extend() definitions for shared setup), /helpers (utility functions — data generators, API clients, assertion helpers), /data (test data factories). Shared reporter config: custom reporter maps test failures to Jira tickets and posts results to Slack. Database cleanup: after each test, API calls remove records created during that test to prevent data accumulation and cross-test contamination.

How do you test multi-tab and multi-window scenarios in Playwright?
▼
Playwright handles multi-tab testing elegantly — unlike Selenium and Cypress which struggle here. Multi-tab test: I listen for the new page event on the browser context simultaneously with triggering the click that opens the new tab: 
const [newPage] = await Promise.all([
  context.waitForEvent('page'),
  page.click('a[target="_blank"]')
]);
await expect(newPage).toHaveURL(/expected-url/);
await expect(newPage.getByText('Expected Content')).toBeVisible();
The key is Promise.all — if you click first then wait, you might miss the page event. For multi-window OAuth popup flows: same pattern — listen for popup, fill credentials in popup window, verify main window receives authenticated state. For Salesforce Service Console specifically, Lightning sub-tabs open within the console frame — I use Playwright's page management APIs to find and interact with the correct tab. Always verify cleanup: after the test, close any opened pages to prevent resource leaks in parallel test workers.
How do you implement visual regression testing with Playwright?
▼
Playwright has built-in visual regression via expect(page).toHaveScreenshot(). On first run it creates baseline screenshots. Subsequent runs compare pixel-by-pixel and fail on differences above the threshold. My implementation strategy: I capture component-level screenshots (not full-page) for targeted regression — expect(page.locator('.status-badge')).toHaveScreenshot('status-badge.png'). This is more stable than full-page screenshots which break on any layout shift anywhere on the page. I configure a diff threshold for anti-aliasing: toHaveScreenshot({maxDiffPixelRatio: 0.01}). For dynamic content (timestamps, user-specific data), I mask those regions before comparison: toHaveScreenshot({mask: [page.locator('.timestamp')]}). Baseline management: screenshots are stored in git alongside tests. When intentional UI changes happen, developers regenerate baselines with --update-snapshots and commit updated screenshots. In CI, visual regression runs on Chromium only — WebKit and Firefox rendering differs enough to cause false failures when compared across browsers.

How do you test file upload and download in Playwright?
▼
File upload: Playwright handles file inputs directly without workarounds. await page.setInputFiles('#file-upload', '/path/to/test-file.pdf') sets the file on a native file input. How do you measure test coverage and what metrics do you track?
▼
I track test coverage across three dimensions. Feature coverage: requirement-to-test traceability matrix — each user story has at least one linked automated test, visible in Jira via JUnit XML integration. Risk-based coverage: not all features need 100% automation. I apply risk weighting — core business flows (case creation, email send, checkout) have 100% automated coverage; low-risk admin screens may have only manual exploratory testing. How do you handle authentication in Playwright — especially for Salesforce applications?
▼
For Salesforce applications, logging in through the UI on every test is slow (Salesforce login takes 5-10 seconds) and fragile. My approach: storageState reuse. I run a one-time setup script that logs in via Playwright and saves the browser session state to a JSON file: await page.context().storageState({path: 'auth/salesforce-agent.json'}). Then every test that needs an authenticated Salesforce agent loads that state instead of logging in: in playwright.config.ts use: {storageState: 'auth/salesforce-agent.json'}. For multiple user roles (agent, admin, customer portal user): I generate separate state files for each role in a global setup file that runs once before all tests. For Salesforce specifically: I also handle session expiry (Salesforce sessions expire after 2 hours by default in sandbox) by checking if the storageState file is stale and regenerating it. For Experience Cloud external users: separate storageState for each external user persona. The result: tests start already authenticated, saving 5-10 seconds per test — critical for large suites with hundreds of tests.

How do you manage environment-specific configuration across dev, staging, and production?
▼
Environment-specific config management is critical — hardcoding staging URLs in test code is a common anti-pattern that causes production incidents. My approach: I use .env files per environment (.env.dev, .env.staging, .env.production) committed to the repo as templates with values redacted, populated by CI/CD secrets at runtime. In Playwright, I access environment variables via process.env.BASE_URL, process.env.API_KEY — never hardcoded strings. In GitHub Actions: I store sensitive values as GitHub Secrets (Settings → Secrets and variables → Actions) and inject them as env vars in the workflow: env: BASE_URL: ${{ secrets.STAGING_URL }}. Different job environments use different secret sets configured per GitHub Environment object. For Azure DevOps: I use Variable Groups linked to different environments — the staging pipeline references the "Staging" variable group, production references "Production." Key principle: the same test code runs identically in staging and production, with only environment variables changing. If I need to skip a test in a specific environment: I use Playwright's test.skip(process.env.ENV === 'production', 'Destructive test') pattern rather than separate test files.
What is Infrastructure as Code and why does a QA engineer need to understand it?
▼
Infrastructure as Code (IaC) means defining infrastructure (servers, databases, networks) using code files — Terraform, AWS CloudFormation, Docker Compose — rather than manual console configuration. Why QA needs it: in modern DevOps teams, QA engineers often need to spin up ephemeral test environments (a fresh environment per PR) rather than sharing a permanent staging environment that accumulates state. IaC enables this — a PR pipeline can provision a complete test environment using Terraform or Docker Compose, run tests against it, then destroy it when done. This eliminates "flaky tests caused by shared staging state" entirely. For my Playwright suite in CI, Docker Compose knowledge means I can define a docker-compose.yml that spins up the application and its dependencies for integration testing in a completely isolated environment. I don't need to be a DevOps expert — but knowing containers, environment variable injection, and service health checks makes me self-sufficient. In the context of Sentinel, I've written the deployment configuration for the platform itself — this IaC mindset directly applies to my automation platform work.
How do you handle secrets and credentials securely in your automation framework?
▼
Credentials in test automation are a real security risk — test accounts sometimes have production data access. My security practices: Never hardcode credentials in code or committed config files. Never log credentials — CI pipeline logs can be searched; I verify that Playwright tests don't log full request headers containing auth tokens. Use least-privilege test accounts — test users have only the minimum permissions needed for their specific test scenarios, not admin access. Store secrets in CI secrets stores (GitHub Secrets, Azure Key Vault) and inject as environment variables at runtime only — never in the YAML file itself. For Salesforce API testing: I use a dedicated Connected App with a test-only profile, not the production integration app credentials. Rotate test credentials quarterly — I document rotation schedules in the team runbook. Session state reuse: I save browser storageState to a file for session reuse, but always add that file to .gitignore — never commit authenticated sessions to git. Any credential leak discovered in git history is treated as a critical incident: immediately revoke and rotate, then use git-filter-repo to clean the history (not just remove from latest commit).

What is trunk-based development and how does it change your testing strategy?
▼
Trunk-Based Development (TBD) means all developers commit directly to the main branch or via very short-lived feature branches (1-2 days max). No long-lived feature branches. Feature Flags control which features are visible in production, decoupling deployment from release. How this changes testing: with TBD, CI must be extremely fast (developers commit multiple times per day) and the test suite must be highly reliable (flaky tests block the entire team, not just one developer's branch). Testing incomplete features: features in development are behind feature flags (disabled in production). I write tests for both flag states — flag OFF (old behavior) and ON (new behavior). This lets unfinished features live in the main branch without breaking production. Continuous testing culture shifts QA's role from gate-keeper to quality enabler — I focus on making the pipeline fast enough (sharding, parallel execution) and reliable enough (zero flaky tests tolerated) that the team can actually commit to trunk without friction. At Horizontal Digital this was the working model — my role was ensuring the CI pipeline quality gate was fast enough and reliable enough that it accelerated rather than blocked delivery.

Business Workflows Reference Guide

The most common business workflows across SFMC and Service Cloud — explained with QA test points at every step
🛒 Abandoned Cart Email Workflow — Most Common E-commerce SFMC Journey
1
Behavior Capture via Web SDK or API
User adds item to cart and leaves without purchasing. E-commerce platform fires a POST to SFMC /interaction/v1/events with ContactKey, CartValue, ProductIDs. QA test: verify API payload schema matches Journey API Event definition exactly; test with missing CartValue field (expect 400); test with ContactKey not in All Subscribers (should fail gracefully).
2
Journey Entry + Deduplication Check
Contact enters Journey. System checks re-entry settings. QA test: fire same API Event twice for same ContactKey within re-entry window — second entry should be rejected (if No Re-entry). Test: contact who abandoned yesterday and today (does the most recent win?).
3
Wait Activity (1 Hour) — Check for Purchase
Journey waits. Meanwhile, another system checks if purchase was completed. QA test in Test Mode: Wait is bypassed. In production sandbox: verify contact exits correctly if purchase completed during wait window. This is the most frequently missed test case.
4
Decision Split — Did They Purchase?
Split evaluates CRM order status or custom DE flag. QA test: contact who purchased goes to True path (exit journey); contact who didn't goes to False path (send email). Test NULL status — where does it go? Should go to False path.
5
Abandoned Cart Email with Product Personalization
AMPscript Lookup pulls product name, image, price from Product DE using ProductID from event payload. QA test: contact with valid ProductID (expect product rendered), invalid ProductID (expect fallback content, not broken email), multiple products in cart (expect all rendered or first N).
6
Engagement Split + Follow-up
Wait 24 hours. Engagement Split: Clicked → exit (conversion achieved); Not Opened → send SMS reminder; Bounced → update contact status to Bounced, exit. QA test all three paths. Verify Bounced path updates _Subscribers status correctly and contact is excluded from future sends.
🏆 Loyalty Program Welcome & Tiered Upgrade Workflow
1
Signup via CloudPage Form
Customer completes loyalty registration form on CloudPage. AMPscript InsertDE() writes record to Loyalty_Members DE and sets initial tier to Bronze. QA test: duplicate email submission (should it update or reject?), missing required field (validation error must display), form submission creates exactly one DE record.
2
Welcome Email — Immediate Triggered Send
Triggered Send fires via API post-form-submission. AMPscript personalizes with FirstName, Tier, PointsBalance (Lookup from Loyalty_Points DE). QA test: FirstName = NULL (expect fallback like "Member"), PointsBalance = 0 (expect "0 points" not empty or error), long FirstName (display truncation).
3
Nightly Points Sync — Automation Studio
Scheduled Automation: SQL Query pulls latest points from CRM Synchronized DE → writes to Loyalty_Points DE (Overwrite mode). QA test: row count after automation matches CRM export count. Test Overwrite mode — previous day's points should be gone. Test what happens if CRM Synchronized DE hasn't updated yet (automation should handle gracefully).
4
Tier Upgrade Decision — Daily Journey Evaluation
Entry source DE has SQL that selects members whose points crossed a tier threshold since yesterday. Decision Split routes: 1000–4999 points → Silver upgrade; 5000+ → Gold upgrade; below → no action. QA test: contact at exactly 1000 points (boundary); contact at 999 (should not upgrade); contact who already upgraded (re-entry prevention). Your Cognizant NFL campaign used this exact pattern.
📦 Transactional Order Confirmation + Post-Purchase NPS Workflow
1
Order Placed → Triggered Send Fires
E-commerce order management system calls SFMC Triggered Send API within seconds of order placement. Payload includes OrderNumber, OrderTotal, ItemList, EstimatedDelivery. QA test: valid order (confirm email received within 2 min), API timeout handling, duplicate order number (idempotency — should not send twice).
2
7-Day Wait → NPS Survey Journey
7 days after order confirmation send, Journey Builder entry source (SQL query on _Sent data view filtered by email subject and send date + 7 days) enters customer into NPS survey journey. QA test: customer who returned order should be excluded from NPS journey. Customer who placed multiple orders in 7 days should only get one NPS (deduplication). Customer who unsubscribed should not receive NPS.
3
NPS Score Capture → CRM Write-Back
Customer clicks score 1–10 in email (link-based NPS). Click event captured in _Click data view. Automation Studio SQL reads _Click data and writes NPS score to CRM via Synchronized DE or API. QA test: verify NPS score writes correctly to CRM Contact record. Test that link URL encoding correctly captures the score value. Test: customer clicks multiple NPS scores (only first should be recorded).
📞 Inbound Case Resolution — Full Lifecycle
1
Case Creation (Web, Email, or Phone)
Case created via web-to-case form, email-to-case, or agent phone logging. Auto-Response Rule fires acknowledgment email. QA test: web-to-case with all required fields, missing required field (validation error), duplicate submission (is a new case created or existing found?). Verify auto-response email contains correct case number and personalization.
2
Assignment Rule Evaluation
Assignment Rule evaluates case fields (Product, Region, Priority, Case Type) and assigns to correct queue. First matching rule wins. QA test: create case matching each rule entry criterion. Critical: create case matching NO criteria — verify fallback queue is assigned, not "unassigned" which disappears from all queue views.
3
Omni-Channel Push to Agent
Omni-Channel pushes case from queue to available agent matching routing config and skills. Agent sees case notification in Service Console widget. QA test: agent at capacity (case should stay in queue), agent without required skill (case should not route to them), all agents offline (case waits in queue — no error thrown).
4
Entitlement Milestone Clock Starts
If account has active entitlement, First Response milestone clock starts at case creation time. Warning fires at 80% of time elapsed. Violation fires at deadline. QA test: shorten milestone to 5 min in sandbox. Verify Warning fires before deadline (email/field update). Verify Violation fires at missed deadline. Test: First Response logged by agent marks milestone Complete — verify violation doesn't fire after completion.
5
Escalation (If Unresolved After SLA)
Escalation Rule fires after configured time: changes Priority to High, reassigns to Tier 2 queue, notifies manager. QA test: shorten escalation time in sandbox. Verify: case reassigned correctly, priority updated, notification email sent with correct case number and details. Test: case resolved before escalation time — escalation should NOT fire.
6
Case Closure + CSAT Survey
Agent closes case. Record-Triggered Flow fires: sends CSAT email via SFMC Triggered Send, marks Resolution milestone Complete, updates Account-level case count metric. QA test: verify CSAT email sends to correct email address (Contact's email, not agent's). Verify customer cannot re-open case without explicit permission. Verify account metrics update correctly after closure.
💬 Live Chat → Case Escalation Workflow
1
Einstein Bot — Initial Triage
Visitor opens chat on website/portal. Einstein Bot greets and attempts to classify intent. Tries to resolve via Knowledge Article suggestion. QA test: intent classification accuracy for all configured intents; bot response to unrecognized input; Knowledge Article returned is relevant to the issue described.
2
Bot Escalates to Human Agent
After N failed resolutions or explicit "speak to agent" request, bot transfers to human via Omni-Channel. Chat transcript passes to agent. QA test: agent receives full chat transcript (not just the last message); extracted entities (order number, email) are pre-populated in case fields; transfer is instantaneous or within SLA.
3
Agent Resolution + Case Creation
Agent resolves chat issue. If case needed: agent creates case from chat record (pre-populated with transcript and customer details). QA test: case creation from chat pre-populates Contact, Subject, Description correctly from chat data. Verify chat-to-case links are maintained for audit trail.
4
Post-Chat Survey
Chat session ends. Automated post-chat survey triggers (built-in Einstein Bot survey or SFMC Triggered Send). QA test: survey sends only when chat had a human agent (not bot-only), survey correctly attributes to the right case and agent, survey score writes back to case record for reporting.
👤 Customer Self-Registration + Self-Service Portal Workflow
1
Guest User Discovers Portal
Anonymous visitor browses public-facing pages (Knowledge Articles, FAQs). Guest User Profile controls what they can see. QA test in incognito: public pages accessible, Salesforce record data NOT visible, DevTools Network tab shows no customer data in API responses.
2
Self-Registration Form
Visitor clicks Register. Fills registration form. Flow creates Contact + External User record. User receives welcome email with login link. QA test: duplicate email address (should update or reject with clear error), weak password (policy enforcement), registration with corporate email vs personal email (any different handling?), welcome email arrives within expected time.
3
Login + Authenticated Session
User logs in. Session established. Portal shows personalized content based on Account/Contact data. QA test: valid credentials, invalid credentials (error message — don't expose whether email exists), account lockout after N attempts, session timeout (re-login required), remember me functionality if offered.
4
Self-Service Case Creation
Authenticated customer creates support case via portal. Case auto-populates Contact from logged-in user. OWD Private ensures customer can only see own cases. QA test: Customer A cannot see Customer B's cases (test as both users in separate browsers simultaneously). Verify case created via portal appears in Service Console for agents. Internal-only Case Comments must NOT appear in portal.
5
Agent Response Visible in Portal
Internal agent responds via Service Console. Response appears as public Case Comment in customer's portal. QA test: public comment appears in portal within expected time. Internal comment (marked Internal) does NOT appear. Case status updates (In Progress → Resolved) reflect in portal view. Notification email to customer triggered by public comment.
6
Customer Self-Closes Case
Customer marks case Resolved via portal. Flow fires: updates Case Status, triggers CSAT survey, updates account self-service metric. QA test: customer can only close their OWN cases (try closing another customer's case via direct URL). Verify re-open is blocked or requires comment. CSAT survey sends correctly.
