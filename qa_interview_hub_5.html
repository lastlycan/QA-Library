🔷 AEM Interview Prep — 100 Questions

50 questions for QA Testers · 50 questions for Developers — Adobe Experience Manager full coverage

🧪 Tester — Architecture Questions
T1What is AEM and describe its major product areas?▼
AEM (Adobe Experience Manager) is an enterprise CMS and DXP. Product areas to know:
  • Sites – Page authoring, editable templates, MSM, Content/Experience Fragments
  • Assets (DAM) – Digital asset management, renditions, Smart Tags, metadata schemas, Brand Portal
  • Forms – Adaptive Forms, Document Services (PDF generation), Form Data Model
  • Commerce – Product catalog integration, CIF (Commerce Integration Framework)
  • Screens – Digital signage delivery
Tie to experience: "I tested Sites and DAM for a global retail brand with 12 regional sites and 2M+ assets."
T2Explain Author, Publish, and Dispatcher — what do you test in each?▼
  • Author – Test dialog fields, component authoring UX, multifield, RTE, page creation, workflow triggers, template selection
  • Publish – Test rendered HTML output, SEO tags, anonymous access, personalised content, correct CSS/JS loading
  • Dispatcher – Test cache invalidation, flush after publish, Dispatcher rules (deny/allow), security headers, vanity URLs, CDN cache behaviour
  • Always test the same page on Author Preview AND Publish — differences reveal replication or rendering bugs
T3How do you verify content replication is working correctly?▼
  • Activate a page and immediately check it on Publish — content visible and matches Author
  • Check Replication Agent queue in Tools → Deployment → Replication — no stuck/error items
  • Test deactivation — page returns 404 on Publish
  • Test replication of assets referenced by page — images, documents accessible on Publish
  • Test tree activation — parent + all child pages replicated together
  • Check replication audit log: /etc/replication/treeactivation
T4How do you use CRXDE Lite during AEM testing?▼
  • Inspect JCR node properties to verify what was saved by a component dialog
  • Check page properties at jcr:content node — title, description, tags, seo fields
  • Verify asset metadata at dam:assetContent node
  • Inspect workflow instances at /etc/workflow/instances
  • Check user/group permissions at /home/users and /home/groups
  • Quickly validate configuration nodes — OSGi config values, Replication Agent settings
T5What is the Sling URL decomposition and why does a tester need to understand it?▼
Sling URLs follow: /content/site/page.selector.extension/suffix
  • Understanding this helps diagnose 404s — is it a missing node, wrong selector, or wrong extension?
  • Test different selectors — .model.json, .infinity.json for API responses
  • Validate extension handling — .html renders page, .pdf triggers document service
  • Suffix tests — verify servlet processes suffix path correctly (e.g. asset download with filename suffix)
  • Key for testing GraphQL and Sling API endpoints used by headless frontends
T6How do you test AEM OSGi configurations?▼
  • Navigate to /system/console/configMgr (Felix Console) to verify config values
  • Test environment-specific configs — Dev/Stage/Prod values correct per runmode (author, publish)
  • Verify externalized configurations — passwords/API keys not hardcoded in content package
  • After deployment, spot-check key OSGi configs: Mail service, Replication agents, Sling referrer filter, Day CQ Link Externalizer
  • Test missing config graceful degradation — service should not crash the entire JVM
💻 Developer — Architecture Questions
D1Explain the JCR (Java Content Repository) and how AEM uses it.▼
JCR is a content repository API (JSR-170/283). AEM uses Apache Jackrabbit Oak as JCR implementation:
  • All content stored as nodes with properties in a tree structure under /content, /apps, /conf, /var
  • Node types: cq:Page, cq:PageContent, dam:Asset, dam:AssetContent, nt:file
  • Oak storage: TarMK (single server), MongoMK (cluster), SegmentNodeStore (AEMaaCS)
  • Indexing via Oak Indexes (Lucene, Property) for query performance
  • Versioning: JCR versioning via mix:versionable — AEM page/asset version history
D2What is Apache Sling and how does it sit on top of the JCR?▼
  • Sling is a RESTful web framework that maps HTTP requests to JCR content nodes
  • Resource resolution: URL → JCR node → Sling Resource → script/servlet selected by resourceType
  • Sling resolves scripts by: resourceType → super resourceType → default (nt:base)
  • Sling Adapters: resource.adaptTo(Page.class), resource.adaptTo(Asset.class)
  • Sling Selectors: different .selector chooses different scripts for same resource
  • SlingHttpServletRequest/Response wrap the JCR Resource for request processing
D3Explain OSGi and how you use it in AEM development.▼
  • OSGi is a modular Java framework — AEM runs on Apache Felix OSGi container
  • Bundles: Java JARs with OSGi metadata — Services, Components registered via @Component annotation
  • @Component + @Service: declare a class as an OSGi service, injectable via @Reference
  • Configuration: @Designate + @ObjectClassDefinition — externalised config via Felix Console / repo config nodes
  • Lifecycle: INSTALLED → RESOLVED → ACTIVE; debug inactive bundles in /system/console/bundles
  • Best practice: prefer @Activate, @Modified, @Deactivate for lifecycle management
D4What is Sling Resource Merger and how does it enable component overlays?▼
  • Resource Merger allows overlaying /libs content from /apps without full copy
  • Place overlay at same path under /apps — only overridden properties/nodes needed
  • sling:hideChildren, sling:hideResource — remove inherited nodes
  • sling:orderBefore — reorder dialog items without full override
  • Key use: overlay Core Component dialog (/apps/core/wcm/components/...) to add custom fields
  • Safer than copying — Core Component updates still apply to non-overlaid parts
D5What are AEM runmodes and how do you use them in configuration?▼
  • Runmodes define the context: author, publish, dev, stage, prod
  • OSGi configs scoped by runmode: /apps/project/config.author, /apps/project/config.publish.prod
  • Sling install folders: /apps/project/install.author — bundles only installed on author
  • Custom runmodes set in sling.properties or start script (-Dsling.run.modes=publish,prod)
  • Test: verify dev config not active in prod — no debug loggers, no test users
  • AEMaaCS: runmodes managed by Cloud Manager — no manual start script
D6Explain ContentBus / Event handling in AEM — how do you listen to JCR events?▼
  • Two main approaches: Sling EventHandler and JCR ObservationManager
  • Sling EventHandler: @Component + EventHandler interface; @Property event.topics = "org/apache/sling/api/resource/Resource/ADDED"
  • Workflow Launcher: UI-based event config — triggers workflow on node create/modify/delete matching a path/node type
  • Sling Jobs: Reliable async processing via JobManager — retry on failure, distributed across cluster
  • Best practice: use Sling Jobs for heavy processing (asset rendition generation, search indexing)
  • Avoid ObservationManager directly — synchronous, blocks the event thread if slow

💻 Developer — Component Questions
D7Explain HTL (Sightly) and how it differs from JSP.▼
  • HTL (HTML Template Language) is AEM's server-side templating language replacing JSP
  • Key advantage: Auto context-aware XSS escaping — output context (HTML, JS, CSS, URI) determines escaping
  • data-sly-use: loads a Sling Model or Use-API class — ${model.title}
  • data-sly-test: conditional rendering — removes element if false
  • data-sly-list / data-sly-repeat: iteration
  • data-sly-resource / data-sly-include: include another resource/script
  • No scriptlets — business logic stays in Sling Models/Use-APIs, not templates
D8Explain Sling Models — annotations, injectors, adaptables.▼
  • @Model(adaptables=..., adapters=...) — declare Sling Model class
  • adaptables: SlingHttpServletRequest (for request-scope data) or Resource (for content-scope)
  • @Inject — generic injector (tries all injectors in order)
  • @ValueMapValue — injects JCR property from current resource's ValueMap
  • @ChildResource — injects child node as Resource
  • @RequestAttribute — injects from request attribute
  • @OSGiService — injects OSGi service into Sling Model
  • @PostConstruct — runs after injection complete, for derived computations
  • @Optional vs @Required — controls null vs injection failure behaviour
D9How do you build a Sling Servlet? When do you use resource-based vs path-based?▼
  • Resource-based: @SlingServlet(resourceTypes="project/components/mycomp", selectors="data", methods="GET") — preferred, security via repository ACLs
  • Path-based: selectors="", paths="/bin/myservlet" — avoid unless necessary; bypass Dispatcher easily; security via Dispatcher rules or Sling Authentication
  • Extend SlingSafeMethodsServlet (GET/HEAD) or SlingAllMethodsServlet (POST/PUT/DELETE)
  • doGet()/doPost() — write response via response.getWriter() or response.getOutputStream()
  • Register with @Component(service=Servlet.class) + OSGi properties for resourceTypes etc.
D10How do you implement a Granite UI dialog with multifield and conditional visibility?▼
  • Dialog defined in: /apps/project/components/mycomp/cq:dialog/content/items/...
  • Multifield: granite:class=coral-Form-field, sling:resourceType=granite/ui/components/coral/foundation/form/multifield; child items define the repeated fields
  • Conditional show/hide: granite:data.showhide-target="#myfield", granite:data.showhide-value="true" on trigger; target field gets granite:class="hide js-hideShowField"
  • Tab layout: sling:resourceType=granite/ui/components/coral/foundation/tabs with child tab items
  • Validation: add required="true", minlength, maxlength as node properties
D11Explain the difference between clientlibs categories and their loading strategy.▼
  • Client Library (clientlib): cq:ClientLibraryFolder node with categories property
  • CSS: loaded in <head> via data-sly-use.clientlib + ${clientlib.css}
  • JS: loaded before </body> via ${clientlib.js}
  • Dependencies: dependencies[] property — ensures dependency loaded before this clientlib
  • Embed: embed[] — merges another clientlib into this one (reduces HTTP requests)
  • AEMaaCS: minification/concatenation automatic in production pipeline
  • Debug: ?debugClientLibs=true disables minification; /libs/granite/ui/content/dumplibs shows all registered clientlibs and their resolved files
D12How do you implement a custom Granite UI widget / custom field type?▼
  • Create a Sling component under /apps that renders the field HTML + clientlib JS
  • Extend granite/ui/components/coral/foundation/form/field as sling:resourceSuperType
  • Client JS: register as a Coral component or use Foundation form utilities to handle save
  • For value persistence: field name must match JCR property name; POST adapter saves via Sling POST Servlet
  • Add renderCondition (granite:rendercondition) to hide field based on resource type or user group
  • Test thoroughly: field value persists after dialog save, loads on dialog re-open, participates in Granite validation

💻 Developer — DAM & Assets
D13How do you write a custom DAM Asset update workflow step?▼
  • Implement WorkflowProcess interface, annotate with @Component(service=WorkflowProcess.class) + "process.label" OSGi property
  • execute(WorkItem, WorkflowSession, MetaDataMap) — main method
  • Get asset path: workItem.getWorkflowData().getPayload() — usually the dam:Asset path
  • Access asset via ResourceResolver from workflowSession.adaptTo(Session.class)
  • Modify asset metadata: asset.getMetadata().put("dc:custom", value)
  • Save session, complete work item via workflowSession.complete(workItem, route)
  • Register in /apps workflow models via CRXDE or workflow model XML
D14How do you implement direct binary upload for AEM Assets (AEMaaCS)?▼
  • AEMaaCS uses direct binary access — client uploads directly to Azure Blob / S3, not through AEM JVM
  • Flow: Initiate upload (POST to /api/assets) → get upload URIs → PUT binary to cloud storage → Complete upload (POST complete URI)
  • Implemented via AEM Upload library (@adobe/aem-upload npm package) or custom HTTP client
  • Benefits: AEM JVM not memory-bottleneck; scales for large files; parallel multi-part upload
  • On-prem alternative: ACS Commons FileTransfer / direct streaming servlet
D15How do you write a custom metadata extraction handler for a proprietary file format?▼
  • Implement com.day.cq.dam.api.handler.AssetHandler or extend BaseExifTool for metadata
  • Or implement Tika Parser if Tika-based — register custom Parser via OSGi
  • Extract metadata properties → write to asset metadata via DamConstants (DC_TITLE, DC_DESCRIPTION)
  • Register handler for MIME type via OSGi property dam.asset.type
  • For AEMaaCS: Asset Compute Service — Node.js worker deployed to Adobe I/O Runtime, processes asset, writes rendition/metadata back to AEM

💻 Developer — Workflows
D19How do you build a custom workflow step with process arguments?▼
  • Implement WorkflowProcess, annotate with @Component(service=WorkflowProcess.class, property={"process.label=My Step"})
  • Process arguments: metaDataMap.get("PROCESS_ARGS", String.class) — configured in workflow model dialog
  • Access page/asset: String path = (String)workItem.getWorkflowData().getPayload(); resourceResolver.getResource(path)
  • Use try-with-resources for ResourceResolver from resourceResolverFactory (service user mapping required)
  • Advance or fail: workflowSession.complete(workItem, workflowSession.getRoutes(workItem,false).get(0))
  • Service users: create system user with minimal permissions; map in org.apache.sling.serviceusermapping
D20How do you implement a Participant step / Dynamic Participant step in a workflow?▼
  • Static Participant: step configured with fixed user/group in workflow model — task sent to that user/group Inbox
  • Dynamic Participant: implement ParticipantStepChooser interface; selectParticipant() returns user/group path based on runtime data (payload, metadata, external service lookup)
  • Example: route approval to regional manager based on page language property
  • Register with @Component(service=ParticipantStepChooser.class, property={"chooser.label=Regional Manager Router"})
  • Assign in workflow model: Dynamic Participant Step → select your chooser
D21What are Sling Jobs and when do you use them instead of workflows?▼
  • Sling Jobs = reliable async task queue; guaranteed delivery, retry on failure, distributed across cluster
  • Use jobs when: Fire-and-forget async processing (send email, index update, cache purge), no user interaction needed, high volume (asset processing), cluster-safe execution
  • Use workflows when: Human participant steps needed, visual audit trail required, complex branching/OR splits
  • JobConsumer: @Component(service=JobConsumer.class, property={JobConsumer.PROPERTY_TOPICS="project/my/topic"})
  • JobManager.addJob(topic, properties) — add job to queue; JobResult.OK / FAILED / ASYNC

D30How do you diagnose and fix AEM memory leaks and OutOfMemoryError?▼
  • Enable heap dump on OOM: -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/path/to/dump.hprof
  • Analyse heap dump with Eclipse MAT or VisualVM — find Retained Heap dominators
  • Common AEM leaks: ResourceResolver not closed (use try-with-resources), Session not closed, event listener not unregistered in @Deactivate
  • Monitor heap via /system/console/memoryusage or JMX (memory pool metrics)
  • Sling Leak Detector: dev tool that alerts on unclosed ResourceResolvers in dev environment
  • Fix: always close ResourceResolver/Session in finally or use IOUtils.closeQuietly
D31How do you implement and optimise Oak Lucene indexes for AEM?▼

  • indexRules: define node types and property paths to index
  • orderedProps: for range queries and sorting (daterange predicate)
  • analyzers: custom text analyzers for full-text search tuning
  • Index-only queries: markIndexOnlyNodes for performance — only indexed data returned
  • Reindex: set reindex=true; Oak reindexes asynchronously (/oak:index/async)
  • Diagnose: QueryDebug UI → "Index Used" column — "none" means no index, add one
  • AEMaaCS: custom indexes must be deployed via Cloud Manager; no ad-hoc reindex
D32How do you secure an AEM implementation against OWASP Top 10?▼
  • Injection: Use QueryBuilder (parameterised); never string-concatenate JCR queries

  • CSRF: Sling Referrer Filter blocks cross-origin POST; AEM uses CSRF token for admin operations
  • Broken Access Control: Deny all in Dispatcher; explicit allow-list per path/method
  • Security Misconfiguration: Remove default admin password; disable /crx/explorer on non-dev; restrict /system/console
  • Vulnerable Components: Subscribe to Adobe Security Bulletins; apply Service Packs promptly
  • SSRF: Validate and whitelist external callout URLs in OSGi config

🤖 Automation Testing

Selenium (Java & Python), Playwright — basic to advanced framework design

Automation Fundamentals
What should and should NOT be automated?▼
  • Automate: Regression tests (stable, repetitive), smoke tests (run per build), data-driven tests (many input combinations), performance baselines, API contract tests
  • Don't automate: Exploratory testing, one-time tests, UI that changes frequently, tests requiring human judgement (UX feel), complex setup with low ROI
  • ROI test: if a test runs <10 times and takes 30 min to automate, don't bother
Explain the Page Object Model (POM) design pattern.▼
  • POM separates page UI interaction code from test logic
  • Each page = one class; locators and methods defined inside
  • Tests call page methods, not raw WebDriver calls
  • Benefits: maintainability (locator changes = one place), readability, reusability
  • Advanced: Page Object + Page Factory (lazy initialisation with @FindBy)
  • Screencast answer: "If the login button moves, I update one locator in LoginPage.java, not 50 test methods."
How do you handle dynamic elements and waits in Selenium?▼
  • Implicit wait – Global timeout; avoid — masks real failures
  • Explicit wait – Wait for specific condition: elementToBeClickable, visibilityOfElement, textToBePresentInElement
  • Fluent wait – Poll at interval, ignore certain exceptions during wait
  • Never use Thread.sleep() — flaky and slow
  • For dynamic IDs: use XPath contains(), starts-with(), or CSS attribute selectors
  • Wait for network idle: Playwright's waitForLoadState('networkidle')
Java + Selenium
Describe your Java Selenium framework structure.▼
  • Structure: src/main/java (page objects, utilities, config), src/test/java (test classes)
  • Test runner: TestNG or JUnit 5 — annotations @Test, @BeforeMethod, @AfterMethod, @DataProvider
  • Build tool: Maven or Gradle — mvn test -Dbrowser=chrome -Denv=staging
  • Reporting: Allure Report — rich HTML reports with screenshots on failure
  • Logging: Log4j2 — debug logs for test steps
  • CI: Jenkins/Azure DevOps pipeline runs mvn test, publishes Allure report
How do you handle browser factory / WebDriver management in Java?▼
  • WebDriverManager (io.github.bonigarcia) — auto-downloads correct ChromeDriver version
  • ThreadLocal<WebDriver> — thread-safe driver instance for parallel execution
  • DriverFactory class — creates driver based on browser param (chrome/firefox/edge)
  • @AfterMethod — quit driver, clear ThreadLocal to prevent memory leak
  • Remote WebDriver — point to Selenium Grid or BrowserStack for cloud execution
Python + Selenium
Describe your Python Selenium framework structure.▼
  • Test runner: pytest — fixtures, parametrize, conftest.py for shared setup
  • Structure: pages/ (POM classes), tests/ (test files), utils/ (helpers, config)
  • Config: pytest.ini or pyproject.toml — markers, base URL, browser param
  • Fixtures: conftest.py @pytest.fixture(scope='session/function') for driver lifecycle
  • Reporting: pytest-html or Allure-pytest — HTML report with failure screenshots
  • Parallel: pytest-xdist (-n auto) — parallel test execution
How do you write a Page Object in Python?▼
Example structure:

class LoginPage:
  USERNAME = (By.ID, "username")
  PASSWORD = (By.ID, "password")
  LOGIN_BTN = (By.CSS_SELECTOR, "button[type='submit']")

  def __init__(self, driver): self.driver = driver

  def login(self, user, pwd):
    self.driver.find_element(*self.USERNAME).send_keys(user)
    self.driver.find_element(*self.PASSWORD).send_keys(pwd)
    self.driver.find_element(*self.LOGIN_BTN).click()
Playwright
What are Playwright's key advantages over Selenium?▼
  • Auto-wait – Playwright auto-waits for elements to be actionable — no explicit waits needed
  • Network interception – Mock API calls, abort requests, modify responses natively
  • Multiple browsers – Chromium, Firefox, WebKit in one framework
  • Traces & Videos – Built-in trace viewer, video recording, screenshot on failure
  • Parallelism – Built-in parallel execution across workers
  • Shadow DOM – Handles shadow DOM without workarounds
  • No flakiness – Retry-ability built in; locators auto-retry until stable
Describe Playwright's locator strategies.▼
  • Role-based: page.get_by_role('button', name='Submit') — accessibility-first
  • Text: page.get_by_text('Welcome') — visible text matching
  • Label: page.get_by_label('Email') — form field by label
  • Placeholder: page.get_by_placeholder('Enter email')
  • Test ID: page.get_by_test_id('submit-btn') — data-testid attribute
  • CSS/XPath: page.locator('button.primary') — as fallback
  • Prefer role/text/label — they reflect user experience, not implementation
How do you use Playwright's API testing features?▼
  • request fixture in Python: async with playwright.request.new_context() as api
  • Make REST calls directly: response = await api.get('/api/users')
  • Assert status: assert response.status == 200
  • JSON body: data = await response.json(); assert data['name'] == 'Alice'
  • Combine with UI: seed test data via API, then test UI — faster than UI setup
  • Mock APIs in browser tests: page.route('**/api/users', handler)
Advanced Automation
How do you design a scalable automation framework from scratch?▼
  • Layer 1 — Core: Driver management, config loader, logger, screenshot utility
  • Layer 2 — Pages: Page Objects per application screen
  • Layer 3 — Components: Reusable component objects (navbar, modal, datepicker)
  • Layer 4 — Tests: Test classes — one file per feature, one method per scenario
  • Data: Externalise test data — JSON/YAML/Excel, environment-specific config
  • Reporting: Allure — steps, attachments, history trend
  • CI: Parallel execution in containers; Slack/email notification on failure
How do you deal with flaky tests?▼
  • Identify with flaky test dashboard — track failure % per test over 30 days
  • Root causes: timing issues (fix with proper waits), test order dependency (make tests independent), environment instability (flag separately)
  • Quarantine flaky tests — move to separate suite, don't block CI
  • Retry policy: 1-2 retries in CI only, not locally — mask real failures if overused
  • Fix flaky tests as bugs — assign in sprint, track resolution rate
